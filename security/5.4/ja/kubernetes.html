Kubernetes

Kubernetesを使用してデプロイする

Kubernetesを使用して、別々のマネージャー、コントローラー、エンフォーサーコンテナをデプロイし、すべての新しいノードにエンフォーサーがデプロイされることを確認できます。SUSE® Securityは、flannel、weave、またはcalicoなどのKubernetesネットワークプラグインを必要とし、サポートします。

サンプルファイルは、1つのマネージャーと3つのコントローラーをデプロイします。すべてのノードにエンフォーサーをデーモンセットとしてデプロイします。デフォルトでは、以下のサンプルはマスターノードにもデプロイされます。

ノードラベルを使用して専用のマネージャーまたはコントローラーノードを指定するには、下部のセクションを参照してください。

セッション状態の問題の可能性があるため、ロードバランサーの背後に1つ以上のマネージャーをデプロイ（スケール）することは推奨されません。SUSE® Securityの設定ファイルのバックアップを保存するためにPersistentVolumeクレームを使用する予定がある場合は、Deploying SUSE® Securityの概要にある一般的なバックアップ/永続データセクションを参照してください。

デプロイメントが統合されたロードバランサーをサポートしている場合は、以下のyamlファイルのコンソールでタイプNodePortをLoadBalancerに変更してください。

SUSE® Securityは、 https://github.com/neuvector/neuvector-helm.にあるHelmチャートを使用したHelmベースのデプロイメントをサポートしています。

OpenShiftの指示用の別のセクションがあり、Kubernetes上のDocker EEにはDockerセクションに記載された特別な手順があります。

SUSE® SecurityのDocker Hub上のイメージ

イメージはSUSE® SecurityのDocker Hubレジストリにあります。マネージャー、コントローラー、エンフォーサーには適切なバージョンタグを使用し、スキャナーとアップデーターにはバージョンを’latest’のままにしてください。次に例を示します。

  • neuvector/manager:5.4.3

  • neuvector/controller:5.4.3

  • neuvector/enforcer:5.4.3

  • neuvector/scanner:latest

  • neuvector/updater:latest

適切なyamlファイル内のイメージ参照を必ず更新してください。

現在のSUSE® Security Helmチャート(v1.8.9+)を使用してデプロイする場合、values.ymlに次の変更を加える必要があります:

  • レジストリをdocker.ioに更新してください。

  • Docker Hubの現在のバージョンに合わせてイメージ名/タグを更新してください。

  • imagePullSecretsは空のままにしてください。

Rancher Manager 2.6.5+ SUSE® Securityチャートからデプロイする場合、イメージはRancher Registryのミラーイメージリポジトリから自動的に取得され、cattle-neuvector-systemネームスペースにデプロイされます。

SUSE® Securityをデプロイする。

  1. SUSE® Securityネームスペースと必要なサービスアカウントを作成する：

    kubectl create namespace neuvector
kubectl create sa controller -n neuvector
kubectl create sa enforcer -n neuvector
kubectl create sa basic -n neuvector
kubectl create sa updater -n neuvector
kubectl create sa scanner -n neuvector
kubectl create sa registry-adapter -n neuvector
kubectl create sa cert-upgrader -n neuvector

  2. (オプション) SUSE® Security Pod Security Admission (PSA)またはPod Security Policy (PSP)を作成する：Kubernetes 1.25+でPod Security Admission（Pod Security Standardsとも呼ばれる）を有効にしている場合、またはKubernetesクラスターで1.25以前のPod Security Policiesを使用している場合、SUSE® Securityに次の内容を追加する（例: nv_psp.yaml）。

    • PSPはKubernetes 1.21で廃止され、1.25で完全に削除されます。

    • ManagerおよびScannerポッドはuidなしで実行される。PSPにルール`Run As User: Rule: MustRunAsNonRoot`がある場合、以下のサンプルyamlに次の内容を追加する（`#`に適切な値を設定する）：

    		 
    securityContext:
    runAsUser: ###

    Kubernetes 1.25+のPSAでは、PSAが有効なクラスターにデプロイするために、SUSE® Securityネームスペースに特権プロファイルのラベルを付ける。

    kubectl label namespace neuvector "pod-security.kubernetes.io/enforce=privileged"

  6. (オプション) フェデレーションマスターおよび/またはリモートマルチクラスター管理サービスを作成する。SUSE® Securityでマルチクラスター管理機能を使用する予定の場合、1つのクラスターにはフェデレーションマスターサービスがデプロイされている必要があり、各リモートクラスターにはフェデレーションワーカーサービスが必要となる。柔軟性のために、各クラスターにマスターサービスとワーカーサービスの両方をデプロイすることを選択でき、任意のクラスターがマスターまたはリモートになることができる。フェデレーテッド クラスター管理

    apiVersion: v1
kind: Service
metadata:
  name: neuvector-service-controller-fed-master
  namespace: neuvector
spec:
  ports:
  - port: 11443
    name: fed
    protocol: TCP
  type: LoadBalancer
  selector:
    app: neuvector-controller-pod

---

apiVersion: v1
kind: Service
metadata:
  name: neuvector-service-controller-fed-worker
  namespace: neuvector
spec:
  ports:
  - port: 10443
    name: fed
    protocol: TCP
  type: LoadBalancer
  selector:
    app: neuvector-controller-pod

    次に、適切なサービスを作成します：

    kubectl create -f nv_master_worker.yaml

  7. プリセットバージョンコマンドを使用して、主要なSUSE® Securityサービスとポッドを作成するか、以下のサンプルyamlを修正してください。プリセットバージョンは、SUSE® Securityコンソールのためにロードバランサーを呼び出します。以下のサンプルyamlファイルを使用する場合は、yamlファイル内のマネージャー、コントローラー、およびエンフォーサーのイメージ参照におけるイメージ名と<version>タグを置き換えてください。また、デプロイメント環境に必要な他の変更（マネージャーアクセスのためのロードバランサー/ノードポート/イングレスなど）を行ってください。以下のYAMLは、v5.4.2以降からデプロイされた場合、内部証明書の変更のために変更する必要があります。このYAMLを参照してください。

    kubectl apply -f https://raw.githubusercontent.com/neuvector/manifests/main/kubernetes/5.4.0/neuvector-k8s.yaml

    または、上記のyamlや以下のサンプルを修正する場合：

    kubectl create -f neuvector.yaml

    それで終わりです！SUSE® Securityコンソールに接続し、admin:adminでログインできるはずです。例えば、`\https://<public-ip>:8443`のように。

neuvector.yamlファイルに指定されたnodeportサービスは、SUSE® Security管理用Webコンソールポートのために、すべてのKubernetesノードでランダムなポートを開きます。代わりに、LoadBalancerまたはIngressを使用して、パブリックIPとデフォルトポート8443を使用することができます。nodeportの場合、そのポートのファイアウォールを通じてアクセスを開く必要がある場合は、必ず行ってください。ホストノードでどのポートが開いているかを確認したい場合は、次のコマンドを実行してください：

kubectl get svc -n neuvector

そして、次のようなものが表示されます：

NAME                          CLUSTER-IP      EXTERNAL-IP   PORT(S)                                          AGE
neuvector-service-webui     10.100.195.99     <nodes>       8443:30257/TCP                                   15m

PKSの変更

PKSはフィールドテスト済みで、プラン/タイルに特権コンテナを有効にする必要があり、Allinone、Controller、EnforcerのためにyamlのhostPathを次のように変更します：

      hostPath:
            path: /var/vcap/sys/run/docker/docker.sock

マスターノードのテイントとトレランス

すべてのテイント情報は、ノードにエンフォーサーをスケジュールするために一致している必要があります。ノード（例：マスター）のテイント情報を確認するには：

kubectl get node taintnodename -o yaml

サンプル出力：

spec:
  taints:
  - effect: NoSchedule
    key: node-role.kubernetes.io/master
  # there may be an extra info for taint as below
  - effect: NoSchedule
    key: mykey
    value: myvalue

上記のように追加のテイントがある場合は、これらをサンプルyamlのトレランスセクションに追加してください：

spec:
  template:
    spec:
      tolerations:
        - effect: NoSchedule
          key: node-role.kubernetes.io/master
        - effect: NoSchedule
          key: node-role.kubernetes.io/control-plane
        # if there is an extra info for taints as above, please add it here. This is required to match all the taint info defined on the taint node. Otherwise, the Enforcer won't deploy on the taint node
        - effect: NoSchedule
          key: mykey
          value: myvalue

マネージャーおよびコントローラー ノードのためのノードラベルの使用

マネージャーとコントローラーがデプロイされるノードを制御するために、各ノードにラベルを付けてください。nodenameを適切なノード名（‘kubectl get nodes’）に置き換えてください。注意:デフォルトでは、Kubernetesはマスターノードにポッドをスケジュールしません。

kubectl label nodes nodename nvcontroller=true

次に、マネージャーとコントローラーのデプロイメントセクションのyamlファイルにnodeSelectorを追加してください。次に例を示します。

          - mountPath: /host/cgroup
              name: cgroup-vol
              readOnly: true
      nodeSelector:
        nvcontroller: "true"
      restartPolicy: Always

専用の管理ノード（監視するアプリケーションコンテナがない場合）である場合、コントローラーノードにエンフォーサーがデプロイされないように、エンフォーサーのyamlセクションにnodeAffinityを追加してください。次に例を示します。

  app: neuvector-enforcer-pod
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
              - matchExpressions:
                - key: nvcontroller
                  operator: NotIn
                  values: ["true"]
      imagePullSecrets:

ローリングアップデート

Kubernetes、RedHat OpenShift、Rancherなどのオーケストレーションツールは、設定可能なポリシーを使用してローリングアップデートをサポートしています。この機能を使用して、SUSE® Security コンテナを更新することができます。最も重要なのは、ポリシー、ログ、および接続データが失われないように、少なくとも 1 つのコントローラー (またはオールインワン) が実行されていることを確認することです。新しいリーダーを選出し、コントローラー間でデータを同期させるために、コンテナを更新する間に最低 120 秒を確保してください。

提供されたサンプルデプロイメント YAML は、ローリングアップデートポリシーをすでに構成しています。SUSE® Security Helm チャートを介して更新している場合は、最新のチャートをプルして、入場制御などの新機能を適切に構成し、SUSE® Security の古いクラスター ロールとクラスター ロール バインディングを削除してください。Kubernetes を介して更新している場合は、以下のサンプルコマンドを使用して手動で新しいバージョンに更新することができます。

サンプル Kubernetes ローリングアップデート

新しいイメージバージョンに更新するだけのアップグレードの場合は、このシンプルなアプローチを使用できます。

デプロイメントまたはデーモンセットがすでに実行中の場合は、YAML ファイルを新しいバージョンに変更し、更新を適用できます。

kubectl apply -f <yaml file>

コマンドラインから SUSE® Security の新しいバージョンに更新するには。

デプロイメントとしてのコントローラー (マネージャーにも適用)

kubectl set image deployment/neuvector-controller-pod neuvector-controller-pod=neuvector/controller:<version> -n neuvector

デーモンセットとしての任意のコンテナの場合：

kubectl set image -n neuvector ds/neuvector-enforcer-pod neuvector-enforcer-pod=neuvector/enforcer:<version>

ローリングアップデートのステータスを確認するには：

kubectl rollout status -n neuvector ds/neuvector-enforcer-pod
kubectl rollout status -n neuvector deployment/neuvector-controller-pod

更新をロールバックするには：

kubectl rollout undo -n neuvector ds/neuvector-enforcer-pod
kubectl rollout undo -n neuvector deployment/neuvector-controller-pod

Kubernetes で REST API を公開する

Kubernetes クラスターの外部からアクセスするために REST API を公開するには、以下のサンプル YAML ファイルを参照してください：
