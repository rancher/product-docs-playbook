kubectlを使用したHPAの管理

このセクションでは、`kubectl`を使用したHPAの管理について説明します。この文書には、次の操作を行うための手順が含まれています。

  • HPAを作成する

  • HPAに関する情報を取得する

  • HPAを削除する

  • CPUまたはメモリの使用率に基づいてHPAをスケールするように設定する

  • Prometheusなどのサードパーティツールを使用してカスタムメトリクスに基づいてHPAをスケールするように設定する

Rancher UIからHPAを作成、表示、削除できます。Rancher UIからCPUまたはメモリの使用量に基づいてスケールするように設定することもできます。詳細については、Rancher UIを使用したHPAの管理を参照してください。CPUまたはメモリ以外のメトリクスに基づいてHPAをスケールするには、依然として`kubectl`が必要です。

HPA管理のための基本的なkubectlコマンド

HPAマニフェストファイルがある場合、`kubectl`を使用してHPAを作成、管理、削除できます。

  • HPAの作成

    • マニフェストを使用して: kubectl create -f <HPA_MANIFEST>

    • マニフェストなし（CPUのみサポート）: kubectl autoscale deployment hello-world --min=2 --max=5 --cpu-percent=50

  • HPA情報の取得

    • 基本: kubectl get hpa hello-world

    • 詳細な説明: kubectl describe hpa hello-world

  • HPAの削除

    • kubectl delete hpa hello-world

HPAマニフェスト定義の例

HPAマニフェストは、`kubectl`を使用してHPAを管理するための設定ファイルです。

以下のスニペットは、HPAマニフェストにおける異なるディレクティブの使用を示しています。各ディレクティブの目的を理解するために、サンプルの下にあるリストを参照してください。

apiVersion: autoscaling/v2beta1
kind: HorizontalPodAutoscaler
metadata:
  name: hello-world
spec:
  scaleTargetRef:
    apiVersion: extensions/v1beta1
    kind: Deployment
    name: hello-world
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      targetAverageUtilization: 50
  - type: Resource
    resource:
      name: memory
      targetAverageValue: 100Mi
ディレクティブ 説明

apiVersion: autoscaling/v2beta1

使用中のKubernetes autoscaling APIグループのバージョン。この例のマニフェストはベータ版を使用しているため、CPUとメモリによるスケーリングが有効になっています。

name: hello-world

HPAが`hello-word`デプロイメントのオートスケーリングを行っていることを示します。

minReplicas: 1

実行中のレプリカの最小数が1未満にならないことを示します。

maxReplicas: 10

デプロイメントの最大レプリカ数が10を超えないことを示します。

targetAverageUtilization: 50

平均実行中のポッドが要求されたCPUの50%以上を使用している場合、ポッドをスケールアップすることを示します。

targetAverageValue: 100Mi

平均実行中のポッドが100Mi以上のメモリを使用している場合、ポッドをスケールアップすることを示します。

Prometheusを使用してカスタムメトリクスでスケールするためにHPAを設定する方法

サードパーティのソフトウェアが提供するカスタムメトリクスに基づいてHPAをオートスケールするように設定できます。サードパーティのソフトウェアを使用したオートスケーリングの最も一般的なユースケースは、アプリケーションレベルのメトリクス（すなわち、1秒あたりのHTTPリクエスト）に基づいています。HPAは`custom.metrics.k8s.io` APIを使用してこれらのメトリクスを取得します。このAPIは、メトリクス収集ソリューションのためにカスタムメトリクスアダプターをデプロイすることで有効になります。

この例では、 Prometheusを使用します。以下の前提から始めます：

  • Prometheusはクラスターにデプロイされています。

  • Prometheusは正しく構成されており、ポッド、ノード、ネームスペースなどから適切なメトリクスを収集しています。

  • Prometheusは以下のURLとポートで公開されています：http://prometheus.mycompany.io:80

PrometheusはRancher v2.0カタログでデプロイ可能です。クラスターでまだ実行されていない場合は、Rancherカタログからデプロイしてください。

HPAがPrometheusからカスタムメトリクスを使用するには、クラスターの`kube-system`ネームスペースに k8s-prometheus-adapterパッケージが必要です。`k8s-prometheus-adapter`をインストールするために、 banzai-chartsで利用可能なHelmチャートを使用します。

  1. クラスターでHelmを初期化します。

      # kubectl -n kube-system create serviceaccount tiller
  kubectl create clusterrolebinding tiller --clusterrole cluster-admin --serviceaccount=kube-system:tiller
  helm init --service-account tiller

  2. GitHubから`banzai-charts`リポジトリをクローンします：

      # git clone https://github.com/banzaicloud/banzai-charts

  3. PrometheusのURLとポート番号を指定して`prometheus-adapter`チャートをインストールします。

      # helm install --name prometheus-adapter banzai-charts/prometheus-adapter --set prometheus.url="http://prometheus.mycompany.io",prometheus.port="80" --namespace kube-system

  4. `prometheus-adapter`が正しく実行されていることを確認します。`kube-system`ネームスペース内のサービスポッドとログを確認します。
