HPAs mit kubectl verwalten

Dieser Abschnitt beschreibt die Verwaltung von HPAs mit kubectl. Dieses Dokument enthält Anweisungen, wie man:

  • Ein HPA erstellen

  • Informationen zu HPAs abrufen

  • Ein HPA löschen

  • Ihre HPAs so konfigurieren, dass sie mit der CPU- oder Speicherauslastung skalieren

  • Ihre HPAs so konfigurieren, dass sie mit benutzerdefinierten Metriken skalieren, wenn Sie ein Drittanbieter-Tool wie Prometheus für Metriken verwenden

Sie können HPAs über die Rancher UI erstellen, anzeigen und löschen. Sie können sie auch so konfigurieren, dass sie basierend auf der CPU- oder Speicherauslastung über die Rancher UI skalieren. Für weitere Informationen siehe HPAs mit der Rancher UI verwalten. Für die Skalierung von HPAs basierend auf anderen Metriken als CPU oder Speicher benötigen Sie weiterhin kubectl.

Grundlegender kubectl-Befehl zur Verwaltung von HPAs

Wenn Sie eine HPA-Manifestdatei haben, können Sie HPAs mit kubectl erstellen, verwalten und löschen:

  • HPA erstellen

    • Mit Manifest: kubectl create -f <HPA_MANIFEST>

    • Ohne Manifest (nur Unterstützung für CPU): kubectl autoscale deployment hello-world --min=2 --max=5 --cpu-percent=50

  • HPA-Informationen abrufen

    • Grundlegend: kubectl get hpa hello-world

    • Detaillierte Beschreibung: kubectl describe hpa hello-world

  • HPA löschen

    • kubectl delete hpa hello-world

Beispiel für die HPA-Manifestdefinition

Das HPA-Manifest ist die Konfigurationsdatei, die zur Verwaltung eines HPA mit kubectl verwendet wird.

Der folgende Ausschnitt zeigt die Verwendung verschiedener Direktiven in einem HPA-Manifest. Siehe die Liste unter dem Beispiel, um den Zweck jeder Direktive zu verstehen.

apiVersion: autoscaling/v2beta1
kind: HorizontalPodAutoscaler
metadata:
  name: hello-world
spec:
  scaleTargetRef:
    apiVersion: extensions/v1beta1
    kind: Deployment
    name: hello-world
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      targetAverageUtilization: 50
  - type: Resource
    resource:
      name: memory
      targetAverageValue: 100Mi
Direktive Beschreibung

apiVersion: autoscaling/v2beta1

Die Version der verwendeten Kubernetes autoscaling API-Gruppe. Dieses Beispielmanifest verwendet die Beta-Version, sodass das Skalieren nach CPU und Speicher aktiviert ist.

name: hello-world

Zeigt an, dass HPA Autoskalierung für die hello-word Implementierung durchführt.

minReplicas: 1

Zeigt an, dass die minimale Anzahl der laufenden Replikate nicht unter 1 fallen kann.

maxReplicas: 10

Zeigt an, dass die maximale Anzahl der Replikate in der Implementierung nicht über 10 steigen kann.

targetAverageUtilization: 50

Zeigt an, dass die Implementierung Pods hochskaliert, wenn der durchschnittliche laufende Pod mehr als 50 % seiner angeforderten CPU verwendet.

targetAverageValue: 100Mi

Zeigt an, dass die Implementierung Pods hochskaliert, wenn der durchschnittliche laufende Pod mehr als 100Mi Speicher verwendet.

Konfiguration von HPA zur Skalierung mit Ressourcenmetriken (CPU und Speicher)

Cluster, die in Rancher v2.0.7 und höher erstellt wurden, haben alle erforderlichen Voraussetzungen (Metrics-Server und Kubernetes-Clusterkonfiguration), um den Horizontal Pod Autoscaler zu verwenden.

Konfiguration von HPA zur Skalierung mit benutzerdefinierten Metriken mit Prometheus

Sie können HPA so konfigurieren, dass es basierend auf benutzerdefinierten Metriken, die von Drittanbieter-Software bereitgestellt werden, automatisch skaliert. Der häufigste Anwendungsfall für Autoskalierung mit Drittanbieter-Software basiert auf Anwendungsebene-Metriken (d.h. HTTP-Anfragen pro Sekunde). HPA verwendet die custom.metrics.k8s.io API, um diese Metriken zu konsumieren. Diese API wird aktiviert, indem ein benutzerdefinierter Metrikenadapter für die Metrik-Sammlungslösung bereitgestellt wird.

Für dieses Beispiel werden wir Prometheus verwenden. Wir beginnen mit den folgenden Annahmen:

  • Prometheus ist im Cluster bereitgestellt.

  • Prometheus ist korrekt konfiguriert und sammelt ordnungsgemäße Metriken von Pods, Knoten, Namespaces usw.

  • Prometheus ist unter der folgenden URL und dem folgenden Port verfügbar: http://prometheus.mycompany.io:80

Prometheus ist im Rancher v2.0 Katalog für die Bereitstellung verfügbar. Stellen Sie es aus dem Rancher-Katalog bereit, wenn es nicht bereits in Ihrem Cluster läuft.

Damit HPA benutzerdefinierte Metriken von Prometheus verwenden kann, ist das Paket k8s-prometheus-adapter im kube-system Namespace Ihres Clusters erforderlich. Um k8s-prometheus-adapter zu installieren, verwenden wir das Helm-Chart, das unter banzai-charts verfügbar ist.

  1. Initialisieren Sie Helm in Ihrem Cluster.

      # kubectl -n kube-system create serviceaccount tiller
  kubectl create clusterrolebinding tiller --clusterrole cluster-admin --serviceaccount=kube-system:tiller
  helm init --service-account tiller

  2. Klonen Sie das banzai-charts Repository von GitHub:

      # git clone https://github.com/banzaicloud/banzai-charts

  3. Installieren Sie das prometheus-adapter Chart und geben Sie die Prometheus-URL sowie die Portnummer an.

      # helm install --name prometheus-adapter banzai-charts/prometheus-adapter --set prometheus.url="http://prometheus.mycompany.io",prometheus.port="80" --namespace kube-system

  5. Überprüfen Sie, ob der Dienst-Pod Running ist. Geben Sie den folgenden Befehl ein. ` # kubectl get pods -n kube-system ` Suchen Sie in der resultierenden Ausgabe nach einem Status von Running. + NAME READY STATUS RESTARTS AGE …​ prometheus-adapter-prometheus-adapter-568674d97f-hbzfx 1/1 Running 0 7h …​ +
