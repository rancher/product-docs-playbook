FAQ（よくある質問と答え）

FAQは定期的に更新され、ユーザーがK3sについて最も頻繁に尋ねる質問に答えるように設計されています。

K3sはKubernetesの適切な代替品ですか？

K3sはCNCF認定のKubernetesディストリビューションであり、標準のKubernetesクラスターに必要なすべての機能を提供できます。それは単により軽量なバージョンです。詳細については、メインドキュメントページをご覧ください。

Traefikの代わりに自分のIngressを使用するにはどうすればよいですか？

K3sサーバーを`--disable=traefik`で起動し、Ingressをデプロイするだけです。

K3sはWindowsをサポートしていますか？

現時点ではK3sはWindowsをネイティブにサポートしていませんが、将来的な対応については前向きに検討しております。

サーバーとエージェントとは具体的に何ですか？

サーバーとエージェントを構成するコンポーネントの詳細については、アーキテクチャページをご覧ください。

ソースからビルドするにはどうすればよいですか？

K3sのhttps://github.com/k3s-io/k3s/blob/main/BUILDING.md[BUILDING.md]を参照してください。

K3sのログはどこにありますか？

K3sのログの場所は、K3sの実行方法やノードのOSによって異なります。

  • コマンドラインから実行すると、ログはstdoutとstderrに送信されます。

  • openrcで実行している場合、ログは`/var/log/k3s.log`に作成されます。

  • Systemdで実行している場合、ログはJournaldに送信され、`journalctl -u k3s`を使用して表示できます。

  • Podのログは`/var/log/pods`にあります。

  • Containerdのログは`/var/lib/rancher/k3s/agent/containerd/containerd.log`にあります。

K3sを起動する際に`--debug`フラグを使用することで、より詳細なログを生成できます（または環境設定ファイル内で`debug: true`を使用します）。

Kubernetesは`klog`というログフレームワークを使用しており、プロセス内のすべてのコンポーネントに対して単一のログ設定を使用します。 K3sはすべてのKubernetesコンポーネントを単一のプロセス内で実行するため、個々のKubernetesコンポーネントに対して異なるログレベルや宛先を設定することはできません。 -v=<level>`または--vmodule=<module>=<level>`コンポーネント引数の使用は、望ましい効果を得られない可能性があります。

さらに多くのログオプションについては、追加のログソースをご覧ください。

K3sをDockerで実行できますか？

はい、K3sをDockerで実行する方法はいくつかあります。詳細については、高度なオプションをご覧ください。

K3sサーバートークンとエージェントトークンの違いは何ですか？

K3sの参加トークンの管理に関する詳細は、`k3s token`コマンドのドキュメントをご覧ください。

異なるバージョンのK3sの互換性はどの程度ですか？

一般的に、https://kubernetes.io/docs/setup/release/version-skew-policy/[Kubernetesバージョンスキューポリシー]が適用されます。

要するに、サーバー側はエージェント側よりも新しいバージョンであっても問題ありませんが、エージェント側がサーバーより新しいバージョンであることはできません。

問題が発生しています。どこで助けを得られますか？

K3sのデプロイに問題がある場合は、次のことを行うべきです：

  1. 既知の問題ページを確認してください。

  2. 追加のOS準備が解決されていることを確認してください。`k3s check-config`を実行し、それが通過することを確認してください。

  3. K3sのhttps://github.com/k3s-io/k3s/issues[問題]とhttps://github.com/k3s-io/k3s/discussions[ディスカッション]を検索して、あなたの問題に一致するものを探してください。

  4. Rancher Slack K3s チャンネルに参加して、サポートを受けてください。

  5. K3s Github に 新しい問題 を提出し、あなたのセットアップと直面している問題を説明してください。
