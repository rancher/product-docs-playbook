k3s-Server

In diesem Abschnitt lernen Sie, wie Sie den K3s-Server konfigurieren.

Bitte beachten Sie, dass Server auch einen Agenten ausführen, sodass alle in der `k3s agent`Dokumentation aufgeführten Konfigurationsoptionen auch auf Servern unterstützt werden.

Optionen sind auf dieser Seite als CLI-Flags dokumentiert, können aber auch als Optionen in der Konfigurationsdatei übergeben werden. Siehe die Dokumentation zur Konfigurationsdatei für weitere Informationen zur Verwendung von YAML-Konfigurationsdateien.

Kritische Konfigurationswerte

Die folgenden Optionen müssen auf allen Servern im Cluster auf denselben Wert gesetzt werden. Sofern dies nicht geschieht, können neue Server dem Cluster nicht beitreten, wenn eingebettetes etcd verwendet wird, oder der Cluster funktioniert nicht korrekt, wenn ein externer Datenspeicher verwendet wird.

--egress-selector-mode Wert

"agent"

Muss eines der folgenden sein:

  • deaktiviert: Der apiserver verwendet keine Agententunnel, um mit Knoten zu kommunizieren. Erfordert, dass Server Agenten ausführen und eine direkte Verbindung zum Kubelet auf den Agenten haben, da der apiserver sonst nicht in der Lage ist, auf Service-Endpunkte zuzugreifen oder kubectl exec und kubectl logs auszuführen.

  • agent: Der apiserver verwendet Agententunnel, um mit Knoten zu kommunizieren. Knoten erlauben die Tunnelverbindung von Loopback-Adressen. Erfordert, dass Server ebenfalls Agenten ausführen, da der apiserver sonst nicht auf Dienstendpunkte zugreifen kann. Der historische Standard für k3s.

  • Pod: Der apiserver verwendet Agententunnel, um mit Knoten und Dienstendpunkten zu kommunizieren, und leitet Verbindungen zu den richtigen Agenten, indem er Knoten überwacht. Knoten erlauben die Tunnelverbindung von Loopback-Adressen oder einem CIDR, das ihrem Knoten zugewiesen ist.

  • Cluster: Der apiserver verwendet Agententunnel, um mit Knoten und Dienstendpunkten zu kommunizieren, und leitet Verbindungen zu den richtigen Agenten, indem er Endpunkte überwacht. Knoten erlauben die Tunnelverbindung von Loopback-Adressen oder dem konfigurierten Cluster-CIDR-Bereich.
