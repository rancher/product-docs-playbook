A

Add-Ons

Dienste über die grundlegenden Komponenten hinaus, die erforderlich sind, um einen Kubernetes-konformen Cluster bereitzustellen, und in zwei Typen kategorisiert:

  • Kern-Add-ons: Add-ons, die erforderlich sind, um einen Kubernetes-konformen Cluster bereitzustellen: DNS, kube-proxy, CNI.

  • Zusätzliche Add-ons: Add-ons, die nicht für einen Kubernetes-konformen Cluster erforderlich sind (z. B. Metrics/Heapster, Dashboard).

Air-Gapped-Umgebung

Einrichten und Ausführen von Kubernetes-Clustern ohne direkten Zugang zum Internet.

B

Bootstrap

Der Prozess, einen Server in einen Kubernetes-Knoten zu verwandeln. Dies kann das Zusammenstellen von Daten umfassen, die beim Erstellen des Servers, der die Maschine unterstützt, bereitgestellt werden müssen, sowie die Laufzeitkonfiguration der Software, die auf diesem Server läuft.

Bootstrap-Cluster

Ein temporärer Cluster, der verwendet wird, um einen Zielverwaltungs-Cluster bereitzustellen.

Bootstrap-Anbieter

Bezieht sich auf einen Anbieter, der eine Lösung für den Bootstrap-Prozess implementiert.

C

CAPI

Core Cluster API

Cluster-API-Anbieter Docker

Cluster-API-Anbieter Google Cloud

CAPIO

Cluster-API-Operator

CAPRKE2

Cluster-API-Anbieter RKE2. Für detaillierte Informationen zu CAPRKE2, siehe die Dokumentation hier.

CAPV

Cluster-API-Anbieter vSphere

CAPZ

Cluster-API-Anbieter Azure

CAPI-Anbieter

Eine öffentliche API, die die Bereitstellung und den Betrieb über die CAPI-Betreiber und Cluster-API Ressourcen erleichtert.

Kind-Cluster

Ein Begriff, der häufig synonym mit dem Workload-Cluster verwendet wird.

Cluster

Eine vollständige Kubernetes-Bereitstellung. Siehe Management-Cluster und Workload-Cluster.

ClusterClass

Eine Sammlung von Vorlagen, die eine Topologie (Steuerungsebene und Arbeiter) definieren, die verwendet wird, um kontinuierlich einen oder mehrere Cluster abzugleichen. Siehe ClusterClass

Cluster-API

Oder Cluster-API-Projekt

Die Cluster-API ist ein Teilprojekt des SIG-Cluster-Lifecycle. Es wird auch verwendet, um die Softwarekomponenten, APIs und die Gemeinschaft, die sie produzieren, zu bezeichnen.

Siehe Kernel-Anbieter

Cluster-API-Operator

Oder Cluster-API-Operator-Projekt

Der Cluster-API-Operator ist ein Teilprojekt des SIG-Cluster-Lifecycle. Es ist darauf ausgelegt, Clusteradministratoren zu befähigen, den Lebenszyklus von Cluster-API-Providern innerhalb eines Management-Clusters mit einem deklarativen Ansatz zu verwalten.

Cluster-API-Provider RKE2

Cluster-API-Provider RKE2 ist eine Kombination aus zwei Anbietertypen: einem Cluster-API-Control-Plane-Provider zur Bereitstellung von Kubernetes-Control-Plane-Knoten und einem Cluster-API-Bootstrap-Provider zum Bootstrapping von Kubernetes auf einer Maschine, auf der RKE2 als Kubernetes-Distro verwendet wird.

Control Plane

Die Menge an Kubernetes-Diensten, die die Grundlage eines Clusters bilden. Siehe auch https://kubernetes.io/docs/concepts/#_kubernetes-control-plane Es gibt zwei Varianten:

  • Selbstbereitgestellt: Ein Kubernetes-Control-Plane, das aus Pods oder Maschinen besteht, die vollständig von einer einzelnen Cluster-API-Implementierung verwaltet wird.

  • Extern oder Verwaltet: Ein Control Plane, das von einem anderen System als der Cluster-API angeboten und kontrolliert wird (z. B. GKE, AKS, EKS, IKS).

Control Plane Provider

Bezieht sich auf einen Provider, der eine Lösung für das Management einer Kubernetes-Control Plane implementiert.

Siehe CAPRRKE2, KCP.

Kernel-Anbieter

Bezieht sich auf einen Anbieter, der die Kerncontroller der Cluster-API implementiert; wenn man bedenkt, dass das erste Projekt, das in einem Management-Cluster bereitgestellt werden muss, die Cluster-API selbst ist, sollte klar sein, warum das Cluster-API-Projekt auch als Kernanbieter bezeichnet wird.

Siehe CAPI.

F

Fleet

Eine Containerverwaltungs- und Implementierungs-Engine, die darauf ausgelegt ist, den Benutzern mehr Kontrolle über den lokalen Cluster und eine ständige Überwachung durch GitOps zu bieten. Werfen Sie einen Blick auf Fleet-Dokumentation, um mehr über Fleet zu erfahren.

I

Infrastruktur-Anbieter

Bezieht sich auf einen Anbieter, der die Bereitstellung von Infrastruktur-/Rechenressourcen implementiert, die vom Cluster oder von Maschinen (z. B. VMs, Netzwerke usw.) benötigt werden. Cloud-Infrastruktur-Anbieter sind AWS, Azure oder Google; während VMware, MAAS oder metal3.io als Bare-Metal-Anbieter definiert werden können.

IPAM-Anbieter

Bezieht sich auf einen Anbieter, der es der Cluster-API ermöglicht, mit IPAM-Lösungen zu interagieren. Die Interaktion des IPAM-Anbieters mit der Cluster-API basiert auf den IPAddressClaim und IPAddress API-Typen.

K

Kubernetes-conformant

Oder Kubernetes-konform

Ein Cluster, das die Kubernetes-Konformitätstests besteht.

Kubernetes Operator

Ein Kubernetes-Operator ist eine Methode zum Verpacken, Bereitstellen und Verwalten einer Kubernetes-Anwendung. Siehe auch https://kubernetes.io/docs/concepts/extend-kubernetes/operator/ für weitere Informationen.

k/k

Bezieht sich auf das Haupt-Kubernetes-Git-Repository oder das Haupt-Kubernetes-Projekt.

KCP

Kubeadm Control-Plane-Provider

M

Computer

Oder Maschinenressource

Die benutzerdefinierte Ressource für Kubernetes, die eine Infrastrukturkomponente darstellt, die einen Kubernetes-Knoten hostet.

Einen Cluster verwalten

Führen Sie Erstellungs-, Skalierungs-, Upgrade- oder Löschvorgänge am Cluster durch.

Verwaltetes Kubernetes

Verwaltetes Kubernetes bezieht sich auf jede Abstraktion zur Bereitstellung und Wartung von Kubernetes-Clustern, die normalerweise als API bereitgestellt wird und nativ in einem Cloud-Anbieter verfügbar ist. Zum Beispiel: EKS, OKE, AKS, GKE, IBM Cloud Kubernetes Service, DOKS und viele mehr im Kubernetes-cloudnativen Ökosystem.

Verwaltete Topologie

Siehe Topologie

Management-Cluster

Der Cluster, in dem ein oder mehrere Infrastruktur-Anbieter betrieben werden und in dem Ressourcen (z. B. Maschinen) gespeichert sind. Typischerweise wird darauf verwiesen, wenn Sie mehrere Workload-Cluster bereitstellen.

N

Knotenpools

Ein Knotenpool ist eine Gruppe von Knoten innerhalb eines Clusters, die alle dieselbe Konfiguration haben.

O

Betriebssystem

Oder OS

Eine allgemein verstandene Kombination aus einem Kernel und einer systemnahen Benutzerschnittstelle, wie Linux oder Windows, im Gegensatz zu einer bestimmten Distribution.

P

Pivot

Pivot ist ein Prozess zum Verschieben der Anbieterkomponenten und der deklarierten Cluster-API-Ressourcen von einem Quell-Management-Cluster zu einem Ziel-Management-Cluster.

Der Pivot-Prozess wird auch verwendet, um einen Management-Cluster zu löschen und könnte auch während eines Upgrades des Management-Clusters verwendet werden.

Anbieter

Oder Cluster-API-Anbieter

Dieser Begriff wurde ursprünglich als Abkürzung für Infrastruktur-Anbieter verwendet, wird jedoch derzeit verwendet, um auf jedes Projekt zu verweisen, das bereitgestellt werden kann und Funktionen für das Cluster-API-Management-Cluster bietet.

Siehe Bootstrap-Anbieter, Control-Plane-Anbieter, Core-Anbieter, Infrastruktur-Anbieter, IPAM-Anbieter, Laufzeit-Erweiterungs-Anbieter.

Anbieterkomponenten

Bezieht sich auf das YAML-Artefakt, das im Rahmen des Veröffentlichungsprozesses für Anbieter veröffentlicht wird; es umfasst normalerweise benutzerdefinierte Ressourcenbeschreibungen (CRDs), Implementierungen (zum Ausführen des Controller-Managers), RBAC usw.

In einigen Fällen wird derselbe Ausdruck verwendet, um auf die Instanzen der oben genannten Komponenten zu verweisen, die in einem Management-Cluster bereitgestellt werden.

Siehe Anbieter-Repository

Anbieter-Repository

Bezieht sich auf den Ort, an dem das YAML für Anbieterkomponenten gehostet wird; normalerweise hostet ein Anbieter-Repository viele Versionen von Anbieterkomponenten, eine für jede veröffentlichte Version.

R

Rancher

Eine Open-Source-https://www.rancher.com/[Plattform], die entwickelt wurde, um die Bereitstellung und Verwaltung von Kubernetes-Clustern zu vereinfachen.

Rancher-Cluster-Agent

Eine Komponente, die von Rancher in jedem Kubernetes-Cluster bereitgestellt wird, den es verwaltet. Seine Hauptaufgabe besteht darin, einen sicheren Kommunikationskanal zwischen dem Rancher-Server und dem Kubernetes-Cluster herzustellen, damit Rancher das Cluster verwalten und mit ihm interagieren kann.

Rancher Manager

Der Rancher Manager (oder Rancher Server) ist der Ort, an dem die Rancher-Benutzeroberfläche und die API gehostet werden, und er kommuniziert mit verwalteten Clustern über Komponenten wie den Rancher Cluster Agent. Er ermöglicht es Benutzern, ihre Kubernetes-Cluster, Anwendungen und Rancher-spezifische Ressourcen wie Kataloge, Benutzer, globale Rollen und mehr zu verwalten.

RKE2

Die nächste Generation von Rancher, eine vollständig konforme Kubernetes-Distribution, die sich auf Sicherheit und Compliance in den USA konzentriert. US-Bundesbehörden liegt. Siehe Dokumentation für weitere Details.

Laufzeit-Erweiterung

Eine externe Komponente, die Teil eines Systems ist, das auf Cluster API aufgebaut ist und Anfragen für einen spezifischen Laufzeit-Hook bearbeiten kann.

Laufzeit-Erweiterung-Anbieter

Bezieht sich auf einen Anbieter, der eine oder mehrere Laufzeit-Erweiterungen implementiert.

S

Skalierung

Sofern nicht anders angegeben, bezieht sich dies auf horizontale Skalierung.

Gestapelter Steuerungsknoten

Ein Steuerungsknoten, bei dem etcd zusammen mit dem Kubernetes API-Server untergebracht ist und als statischer Pod läuft.

Server

Die Infrastruktur, die einen Machine Resource unterstützt, typischerweise entweder eine Cloud-Instanz, eine virtuelle Maschine oder einen physischen Host.

SUSE® Rancher Prime Cluster API

Ein Kubernetes-Operator, der die Integration zwischen Rancher Manager und Cluster API (CAPI) ermöglicht, mit dem Ziel, vollständige CAPI-Unterstützung für Rancher bereitzustellen.

T

Topologie

Ein Feld in der Cluster-Objektspezifikation, das es ermöglicht, die Form der Steuerungsebene und der Arbeitsknoten des Clusters von einem einzigen Kontrollpunkt aus zu definieren und zu verwalten. Die Topologie des Clusters basiert auf einer ClusterClass. Manchmal wird es auch als verwaltete Topologie bezeichnet.

Siehe ClusterClass

Schildkröten

Bezieht sich auf SUSE® Rancher Prime Cluster API

W

Workload-Cluster

Ein Cluster, das von einem ClusterAPI-Controller erstellt wurde, der kein Bootstrap-Cluster ist und für Endbenutzer gedacht ist, im Gegensatz zu CAPI-Tools.

WorkerClass

Eine Sammlung von Vorlagen, die eine Reihe von Arbeitsknoten im Cluster definieren. Eine ClusterClass enthält null oder mehr WorkerClass-Definitionen.

Siehe ClusterClass
